# Codex 接入 GPT 模型

本教程介绍如何下载安装 Codex，并使用 SurplusToken API Key 接入 GPT 模型。配置完成后，可以在 Codex CLI 与 Codex Desktop 中使用 SurplusToken 提供的模型服务。

:::warning 密钥安全
API Key 等同于账号密码。不要把密钥发送给他人，不要写入项目源码或提交到 Git，也不要在公开截图、终端录屏和日志中展示完整密钥。

下文中的 `<SURPLUSTOKEN_API_KEY>` 是密钥占位符，请替换为你自己的 API Key。
:::

## 开始前准备

开始配置前，请先完成以下操作：

1. 登录 [SurplusToken](https://surplustoken.com)。
2. 打开 **API 密钥**页面，点击右上角的**创建密钥**。
3. 输入便于识别的名称，例如 `codex-plus-team`。
4. 在**分组**中选择 `plus/team`。该分组用于 Plus 和 Team 账号，页面当前显示倍率为 `0.15x`；详细规则参见 [plus/team 分组说明](/groups/plus-team)。
5. 点击**创建**，复制新生成的 API Key 并妥善保存。
6. 确认账户余额充足，并确认该分组支持准备使用的模型。

选择分组时，确认选中的是 `plus/team`：

/>

完成选择后的表单如下。额度限制、速率限制、有效期等高级选项可以按需设置；首次接入可以保持默认值：

<img

:::tip 分组文档与交叉引用
不同分组的适用账号、倍率、可用模型和限制可能不同。本教程只说明 Codex 的接入流程；选择其他分组前，请先阅读对应的[分组说明](/groups/)。
:::

SurplusToken 的公网 API Base URL 为：

```text
https://surplustoken.com/v1
```

如果平台控制台提供了校内专用地址，请以控制台显示的地址为准。

## 1. 下载并安装 Codex

<SystemContent system="windows">

### Windows

#### 方式一：Codex Desktop

访问 [Codex 官方页面](https://openai.com/zh-Hans-CN/codex/)，下载并安装 Windows 版 Codex Desktop。

Codex Desktop 可以直接使用原生 Windows 环境。如果安装或沙箱初始化受到学校、公司设备策略限制，也可以改用后文的 WSL2 方式。

#### 方式二：原生 Windows CLI

先安装当前 LTS 版本的 [Node.js](https://nodejs.org/)，然后打开 PowerShell 或 Windows Terminal：

```powershell
npm install -g @openai/codex
codex --version
```

#### 方式三：WSL2 CLI

如需在 Linux 开发环境中使用 Codex，请以管理员身份打开 PowerShell 或 Windows Terminal：

```powershell
wsl --install
```

安装完成后重启电脑，打开 Ubuntu 终端：

```bash
curl -fsSL https://chatgpt.com/codex/install.sh | sh
codex --version
```

使用 WSL2 时，后文中的 `~/.codex` 指 WSL 用户目录中的配置目录，不是 Windows 的 `%USERPROFILE%\.codex`。

</SystemContent>

## 2. 找到 Codex 配置目录

Codex 的用户级主配置文件是 `config.toml`，API Key 登录信息保存在 `auth.json` 中：

| 运行环境       | 主配置                             | API Key                          |
| -------------- | ---------------------------------- | -------------------------------- |
| Windows 原生   | `%USERPROFILE%\.codex\config.toml` | `%USERPROFILE%\.codex\auth.json` |
| Windows + WSL2 | `~/.codex/config.toml`             | `~/.codex/auth.json`             |
| macOS          | `~/.codex/config.toml`             | `~/.codex/auth.json`             |
| Linux          | `~/.codex/config.toml`             | `~/.codex/auth.json`             |

<SystemContent system="windows">

### Windows 原生环境

打开 PowerShell，创建配置目录并打开：

```powershell
New-Item -ItemType Directory -Force "$env:USERPROFILE\.codex" | Out-Null
explorer "$env:USERPROFILE\.codex"
```

如果使用 WSL2，请在 Ubuntu 终端中创建配置目录：

```bash
mkdir -p ~/.codex
chmod 700 ~/.codex
```

原生 Windows 与 WSL2 是两个独立环境。在哪个环境中运行 Codex，就配置哪个环境中的 `.codex` 目录。

</SystemContent>

<SystemContent system="macos">

在 Terminal 中创建配置目录：

```bash

chmod 700 ~/.codex
```

</SystemContent>

<SystemContent system="linux">

在终端中创建配置目录：

## 3. 配置 GPT 模型

在 Codex 配置目录中创建或打开 `config.toml`，写入：

```toml
model_provider = "surplustoken"
model = "gpt-5.6-sol"
model_reasoning_effort = "high"
disable_response_storage = true

[model_providers.surplustoken]
name = "SurplusToken"
base_url = "https://surplustoken.com/v1"
wire_api = "responses"
requires_openai_auth = true
```

这份配置会让 Codex：

- 使用名为 `surplustoken` 的自定义模型提供商；
- 默认调用 `gpt-5.6-sol`；
- 通过 OpenAI Responses API 兼容协议发送请求；
- 使用 API Key 进行身份验证。

:::info Base URL 填写规则
`base_url` 应填写到 `/v1`，不要写成 `/v1/responses`。Codex 会自动拼接 Responses API 的请求路径。

如果使用平台提供的校内专用地址，只替换 `base_url` 的值，其他字段保持不变。
:::

:::tip TOML 字段位置
`model_provider`、`model`、`model_reasoning_effort` 和 `disable_response_storage` 必须放在文件中第一个 `[表名]` 之前。

从 `[model_providers.surplustoken]` 开始，后续普通字段会被视为该配置表的成员，不再属于 Codex 顶层配置。
:::

## 4. 登录 SurplusToken API Key

推荐让 Codex 从标准输入保存 API Key，避免密钥出现在命令历史中。

<SystemContent system="windows">

### Windows 原生环境

在 PowerShell 中运行：

```powershell
$env:OPENAI_API_KEY = Read-Host "SurplusToken API Key"
$env:OPENAI_API_KEY | codex login --with-api-key
Remove-Item Env:OPENAI_API_KEY
```

### Windows + WSL2

在 Ubuntu 终端中运行：

```bash
read -rsp "SurplusToken API Key: " SURPLUSTOKEN_API_KEY
echo
printf '%s' "$SURPLUSTOKEN_API_KEY" | codex login --with-api-key
unset SURPLUSTOKEN_API_KEY
chmod 600 ~/.codex/auth.json
```

</SystemContent>

<SystemContent system="macos">

在 Terminal 中运行：

```bash
read -rsp "SurplusToken API Key: " SURPLUSTOKEN_API_KEY
echo
printf '%s' "$SURPLUSTOKEN_API_KEY" | codex login --with-api-key
unset SURPLUSTOKEN_API_KEY
chmod 600 ~/.codex/auth.json
```

</SystemContent>

<SystemContent system="linux">

在终端中运行：

```bash
read -rsp "SurplusToken API Key: " SURPLUSTOKEN_API_KEY
echo
printf '%s' "$SURPLUSTOKEN_API_KEY" | codex login --with-api-key
unset SURPLUSTOKEN_API_KEY
chmod 600 ~/.codex/auth.json
```

</SystemContent>

登录完成后检查状态：

```bash
codex login status
```

如果需要手动创建 `auth.json`，文件内容如下：

```json
{
  "OPENAI_API_KEY": "<SURPLUSTOKEN_API_KEY>"
}
```

:::warning
`auth.json` 必须是合法 JSON，不能包含注释，最后一个字段后不能有逗号。
:::

## 切换 Provider 后找回历史记录

切换模型提供商通常不会删除 Codex 保存在本机的会话。历史记录暂时看不到时，常见原因是启动 Codex 的项目目录、Windows/WSL 环境或 `CODEX_HOME` 与以前不同。

先进入原来使用 Codex 的项目目录，然后运行：

```bash
codex resume --all
```

`codex resume` 默认打开会话选择器，`--all` 会把当前目录以外的本机会话也列出来。已经进入 Codex 时，也可以输入 `/resume`。

仍然找不到时，可以在 Codex 中粘贴下面的只读排查 Prompt：

```text
请只做只读检查，不要修改、移动或删除任何文件。检查当前 Codex 使用的 CODEX_HOME、用户级 config.toml、history.jsonl 和 sessions 目录，确认旧会话是否仍在本机；检查我是否从不同的项目目录、原生 Windows/WSL 环境或不同的 CODEX_HOME 启动了 Codex。请列出发现的问题和安全的恢复命令，优先使用 codex resume --all，不要重写会话文件，也不要显示 API Key。
```

:::warning
不要为了“恢复历史”手动合并、改名或覆盖 `history.jsonl` 与 `sessions` 中的文件。先备份整个 `.codex` 目录，再处理任何文件级迁移。
:::

## 使用 Prompt 改为 SurplusToken Provider

如果原来使用其他 Provider，可以让 Codex 在确认路径并备份后修改用户级配置。请粘贴以下 Prompt：

````text
请把当前用户级 Codex 配置切换到名为 surplustoken 的自定义模型提供商，并遵守以下要求：

1. 先确认当前运行环境实际使用的 CODEX_HOME 和用户级 config.toml 路径。Windows 原生通常是 %USERPROFILE%\.codex\config.toml，macOS、Linux 和 WSL 通常是 ~/.codex/config.toml。
2. 只修改用户级 config.toml，不要在项目的 .codex/config.toml 中配置 model_provider 或 model_providers。
3. 修改前创建带时间戳的备份；保留所有无关配置，不要覆盖原文件中的其他表。
4. 将以下顶层字段放在文件第一个 [表名] 之前，并添加或更新 provider 表：

```toml
model_provider = "surplustoken"
model = "gpt-5.6-sol"
model_reasoning_effort = "high"
disable_response_storage = true

[model_providers.surplustoken]
name = "SurplusToken"
base_url = "https://surplustoken.com/v1"
wire_api = "responses"
requires_openai_auth = true
```

5. 不要读取、显示或修改 API Key，不要改动 auth.json。
6. 修改后验证 TOML 语法，展示不含敏感信息的变更摘要，并提醒我完全重启 Codex 后使用 /status 验证 provider 和模型。
````

如果尚未保存 SurplusToken API Key，再按本文第 4 节运行 `codex login --with-api-key`。Provider 定义属于用户级配置；项目级 `.codex/config.toml` 不能覆盖这类设置。

## 常见问题

### 返回 401 Unauthorized

通常表示 API Key 不正确、已撤销或已过期：

- 重新复制完整密钥，检查前后是否有空格或换行；
- 运行 `codex login status` 检查登录状态；
- 确认修改的是当前运行环境实际读取的 `.codex` 目录；
- 重新运行 `codex login --with-api-key` 保存密钥。

### 返回 404 或提示模型不存在

依次检查：

- `base_url` 是否以 `/v1` 结尾；
- 是否错误写成了 `/v1/responses`；
- `model` 是否与平台提供的模型 ID 完全一致；
- 当前 API Key 绑定的分组是否支持该模型。

### 返回 429 Too Many Requests

可能是请求频率、并发、余额、订阅额度或上游账号受限。请查看 SurplusToken 控制台中的密钥状态、账户余额和用量记录，稍后重试。

### 修改配置后没有生效

- 在 Codex 中输入 `/status`，确认实际读取的模型和提供商；
- 检查顶层 TOML 字段是否错误写在 `[model_providers.surplustoken]` 后面；
- 检查是否存在优先级更高的项目配置 `.codex/config.toml`；
- 完全退出并重新打开 Codex CLI 或 Codex Desktop。

### CLI 可用，但 Desktop 不可用

Windows 用户应先确认 Desktop 当前选择的是原生 Windows 还是 WSL 环境，再检查对应环境中的 `.codex` 目录。macOS 用户应确认 Desktop 与 CLI 使用的是同一个系统账号。修改配置后完全退出并重启 Desktop。

## 撤销或更换密钥

怀疑 API Key 泄露时：

1. 立即在 SurplusToken 平台撤销旧密钥。
2. 创建新密钥，并选择正确的分组。
3. 重新运行 `codex login --with-api-key` 保存新密钥。
4. 重启 Codex，并重新发起测试请求。
